# PM_RULES（Minimal Director Mode 对齐版）

> 适用范围：`AI-music-producer PRD_v1.1.md` 当前版本
> 审核目标：让 PM 在不看实现细节的前提下，也能稳定做出“可上线 / 不可上线”的一致判定

---

## 0. PM 角色定义

PM 不是“流程记录员”，而是本系统的质量闸门与审美边界守卫。

PM 只关心三件事：
1. 是否严格遵守 PRD 的最终态架构。
2. 是否具备真实可回放的证据链。
3. 是否达到“可投喂 Suno/MiniMax”的工业交付标准。

---

## 1. One Law（四条缺一不可）

任何 PR 合并前，以下四条必须全部满足：

1. **需求追溯**：变更必须可追溯到 PRD 条款。  
2. **测试证明**：必须有可复现测试证据（至少 Green 证据，关键改动需有 Red->Green 说明）。  
3. **无冗余制造**：不得引入重复脚本、重复流程、无必要中间层。  
4. **Gate 全通**：CI 与关键质量门全绿。  

任一不满足：**直接驳回**。

---

## 2. 架构一致性红线（与 PRD 强绑定）

### 2.1 模型基座红线
- LLM 基座必须是 **Kimi 2.5**（唯一）。
- 若出现 OpenAI/GPT/Claude 的生产路径配置，判为架构偏离。

### 2.2 流程调度红线
- 主链必须是 **8 步 Python DAG 固定顺序**。
- 大模型只允许用于：
  1) 意图解析
  2) 一次性歌词生成
- 严禁把工具调度权交给大模型。

### 2.3 生成策略红线
- 填词为“三维边界软提示”策略（物理边界 + 审美红线 + 意境引导）。
- 禁止回流到复杂硬约束引擎（POS/向量/SVO/motif 主轴邻近度）。

### 2.4 物理拦截红线
- 只能有且必须有三道后置规则：
  1) 副歌末字开口音校验
  2) 烂梗黑名单正则校验
  3) 字数兜底校验
- 任一道失败，必须有明确重试或中止证据。

### 2.5 数据管理红线
- 采用 Session 临时目录 + 隐藏日志目录。
- 24h 自动清理生效。
- 终端不得泄露中间 JSON/raw response/traceback。

---

## 3. G0-G8 审核门禁

### G0 环境与入口
- 运行环境符合 Windows 原生 CLI。
- 不引入 Docker 作为默认依赖。

### G1 变更范围与提交质量
- 提交信息格式：`type(scope): summary`。
- 单次修复聚焦，不混入无关改动。

### G2 失败证据（必要时）
- 对 bugfix/质量修复，需能说明失败根因。
- 至少包含一次明确失败现象与触发条件。

### G3 通过证据
- 本地与 CI 至少一条一致口径的通过证据。
- 关键链路测试可复现。

### G4 文档一致性
- PRD、PM 规则、证据包字段命名不冲突。
- 交付文件路径与规则一致。

### G5 Hook 与脚本质量
- pre-commit/pre-push 逻辑不应误杀必要产物。
- 禁止常态化 `--no-verify`。

### G6 CI 质量门
- `ci-quality-gates` 必须成功。
- 若 CI 失败且本地成功，必须提供环境差异解释。

### G7 证据链闭环
- 审计日志需可回放：run_id/trace_id/event/decision/reason_code。
- 不接受仅口头“已完成”。

### G8 Show-Me-Output
- 必须看到真实终局产物（可投喂文本），不是仅展示过程日志。
- 需要可定位到同一运行会话。

---

## 4. 交付物硬性要求

按当前规则，审核优先看根目录交付：

1. `OUTPUT_DEMO_PROMPT.md`（当 score >= 9）
2. `PM_AUDIT_REPORT.md`（当 score < 9）

两者可在 `docs/pm/` 保留镜像，但根目录为 PM 审核入口。

---

## 5. 审核判定矩阵

### 5.1 直接通过（Go）
同时满足：
- One Law 四条全满足
- G0-G8 全绿
- CI 成功且关键产物可复核

### 5.2 继续修订（Request Changes）
任一触发：
- CI 失败
- 证据链断裂（无 run_id/trace_id 或日志不可回放）
- 架构偏离（非 Kimi 2.5、非 8 步 DAG、越权调度）
- 终端/交付不符合规范

### 5.3 紧急驳回（No-Go）
任一触发：
- 使用 mock/placeholder 伪造通过
- 隐瞒失败路径或篡改证据
- 输出不具备投喂价值且无修复计划

## 6. PM 评论模板（标准化）

### 6.1 通过模板
```text
【PM 复审结论】Go（同意转 Ready）

依据：One Law 四条满足，G0-G8 全绿，ci-quality-gates 成功，交付物与证据链完整。
请保持后续提交流程：单一可验收 SHA + CI 链接 + 关键证据摘要。
```

### 6.2 驳回模板
```text
【PM 复审结论】继续修订（Request Changes）

阻断项：<列 P0>
证据：<run/sha/log/文件路径>
通过条件：<下一次可验收清单>
```

---

## 7. PM 审核行为准则

1. 不给悬空结论：结论必须附证据与下一步。
2. 不做“口头通过”：通过必须可追溯到具体 SHA 与 CI run。
3. 不替开发拍脑袋：必须下发可执行任务单（P0/P1、文件路径、验收命令、通过标准）。
4. 不牺牲审美底线：可运行不等于可发布，需兼顾可唱性与文本质量。
