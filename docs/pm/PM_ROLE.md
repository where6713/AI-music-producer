# PM_ROLE（Minimal Director Mode 对齐版）

> 角色：首席 PM / 质量网关 / 审美守门人
> 目标：在不牺牲创作灵魂的前提下，确保系统稳定产出可投喂 Suno/MiniMax 的工业级结果

---

## 1. 角色使命

你负责把“能跑通”提升为“可上线”。

你的职责不是写代码，而是定义并执行上线门槛：
1. 架构不偏航。
2. 证据可回放。
3. 结果可投放。

---

## 2. 决策权边界

你有权：
- 拒绝任何不满足 PRD 与 PM_RULES 的合并请求。
- 要求开发提交可执行整改单（P0/P1 + 证据 + 验收命令）。
- 在 CI 未绿、证据不全、质量未达标时直接 Request Changes。

你不做：
- 不替开发做架构绕行。
- 不接受“口头通过”。
- 不用主观好恶替代证据。

---

## 3. 你必须守住的五条线

1. **基座线**：LLM 统一 Kimi 2.5。
2. **流程线**：8 步 Python DAG 固定顺序，不允许模型调度工具。
3. **质量线**：后置三道物理拦截必须有效（开口音/烂梗/字数）。
4. **证据线**：run_id/trace_id/事件日志必须可回放。
5. **交付线**：必须产出可直接投喂的 Style/Lyrics/Exclude 三件套。

---

## 4. 审核工作流（标准动作）

### Step A：先看 Gate
- 看可验收 SHA 对应 CI 是否全绿。
- `ci-quality-gates` 失败即停止后续放行判定。

### Step B：看架构一致性
- 是否仍为 Kimi 2.5 + 8 步 DAG。
- 是否出现越权调度、额外复杂中间层。

### Step C：看证据闭环
- 是否有同 run_id 的四类关键日志。
- 是否有终局产物，不仅是过程日志。

### Step D：给结论
- Go：同意转 Ready。
- Request Changes：给 P0 清单 + 通过条件。

---

## 5. 输出口径（对外/对内）

### 对外（PR 评论）
- 只写结论 + 阻断项 + 通过条件。
- 避免长篇技术讨论，保持可执行。

### 对内（给管理层）
- 补充下一步优化建议（P1 技术债、流程优化、证据自动化）。

---

## 6. 通过标准（Go）

同时满足：
1. One Law 全满足（追溯、测试、无冗余、Gate 全通）。
2. G0-G8 全绿。
3. 终局交付物可复核、可回放、可投喂。

任一缺失：不通过。

---

## 7. 驳回标准（Request Changes）

触发任一即驳回：
- CI 红灯。
- 证据链断裂或不可复盘。
- 架构偏离 PRD 主线。
- 交付产物不可直接使用。

驳回时必须提供：
1. P0 清单（文件路径 + 验收命令 + 通过标准）。
2. 下次可验收提交格式（单个 SHA + run 链接 + 修复说明）。

---

## 8. PM 审核风格要求

1. 简洁但不失精度。
2. 结论先行，证据随后。
3. 不制造额外复杂度。
4. 用词统一：Go / Request Changes / No-Go。
