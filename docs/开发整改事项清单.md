# 开发整改事项清单（PM 强制执行）

> 适用范围：`AI-music-producer` 全体开发成员
> 执行标准：`PM_RULES + PRD + one law` 三基线一致
> 口径：证据优先、真实数据、真实链路、可复现

---

## A. 最高优先级（P0，必须先完成）

1. 修复主链路命令可用性
- 目标：`apps/cli/main.py` 与 `src/main.py` 的命令链路稳定可执行，不再出现参数解析崩溃。
- 验收：真实场景命令可执行并生成三件套与追溯文件。
- 证据：命令、返回码、产物路径。

2. 打通模型提供方适配层（Anthropic / OpenAI-compatible）
- 目标：在不违反 PRD 约束前提下，主链路支持当前环境可用模型通道。
- 验收：`.env` 中可用 key 能驱动同一主链路完成生成。
- 证据：`trace.json` 包含 model_used、token usage、llm_calls。

3. 落地 PRD v2.1 Step1：retriever + corpus 注入
- 目标：实现 `src/retriever.py`，按 emotion_tags + valence 检索 2-3 条示例注入。
- 验收：`lyric_payload.json` 出现 `few_shot_examples_used` 且数量 >= 2。
- 证据：示例 source_id、title、matched tags。

4. 落地 PRD v2.1 Step2/3：三变体生成与排序选优
- 目标：输出 3 个 variants，lint 对三变体评分排序并给出 `chosen_variant_id`。
- 验收：`variants` 恰好 3 个，POV 各不相同，存在 rank。
- 证据：`lyric_payload.json` + `trace.json` + lint 报告。

5. 落地 PRD v2.1 Step4：真实 Targeted Revise
- 目标：Top-1 违规时触发一次真实定向重写，不允许仅写 prompt 到 trace。
- 验收：触发场景下 llm_calls 从 1 变 2，违规项减少或清零。
- 证据：before/after lint 对比、trace 中 revise 记录。

---

## B. 高优先级（P1，本周闭环）

6. 强化 G4 失败可观测性
- 目标：`docs-alignment-check` fail 时打印 `failed_checks`。
- 验收：缺失 `manifest_path` 时 CLI 显式输出 `failed_checks: manifest_path`。

7. 增加真实 E2E 回归用例
- 目标：新增 1 条“真实路径 + 真实意图 + 真实产物”回归流程。
- 验收：固定命令可复跑并产出 `out/*` 五件套。

8. 完善 G3 输出一致性判定
- 目标：避免关键词误判（如“not pass”被当 pass）。
- 验收：增加反例测试并通过。

9. 统一审计模板自动化
- 目标：每个 Gate PR 自动生成四段审计骨架与命令输出归档。
- 验收：新 PR 默认带模板，不再手工拼接。

---

## C. 工程质量（P2，持续改进）

10. 建立“禁止旁路生成”守卫
- 目标：CI 或 hooks 检测旁路脚本调用模型行为（规则白名单）。
- 验收：发现旁路调用时阻断并给出定位信息。

11. 建立“语义可解释性”人工抽检标准
- 目标：针对歌词可理解性、意象一致性、句间因果建立人工审稿打分表。
- 验收：每次版本至少 3 首样本盲评记录。

12. 版本化语料质量基线
- 目标：corpus 每次更新均附统计摘要（标签覆盖、长度分布、重复率）。
- 验收：语料 PR 必带统计与回滚点。

---

## 交付要求（逐条关闭）

每条整改项必须在 PR 描述中包含：

1. 变更文件路径
2. 对应测试用例
3. 执行命令与结果
4. 回滚方案

未满足上述 4 项，视为未完成。
