# 审计调度机制

> 目的：用固定节奏 + 风险触发两条线，持续检查安全、冗余和越权问题。

## 1. 审计类型与频率

### A) 节奏型审计（固定）
- 每完成 **3 个模块**：触发一次代码结构与冗余审计
- 每周一次：触发安全与权限审计
- 每次准备合并主线前：触发合并前综合审计

### B) 事件型审计（触发）
- 出现权限相关改动（鉴权、路径访问、敏感配置）
- 出现大范围重构（跨 3 个以上目录）
- 出现测试异常波动（通过率显著下降）

## 2. 审计项清单

### 安全审计
- 越权访问风险（是否绕过预期权限路径）
- 敏感信息暴露（密钥、令牌、私钥痕迹）
- 危险命令/危险脚本链路

### 冗余审计
- 重复脚本功能（同义不同名）
- 重复文档权威定义
- 低价值重复逻辑

### 质量审计
- 是否存在“只为过测试”的表演性改动
- 是否存在计划外扩写
- 是否满足单一职责边界

## 3. 审计输出格式（必填）
- 审计轮次：
- 触发原因：
- 审计范围（文件/模块）：
- 发现项（阻断 / 警告 / 建议）：
- 处置动作与责任人：
- 回归验证结论：

## 4. 处置规则
- 阻断项：禁止进入下一 Gate，必须修复并复审
- 警告项：允许继续，但需登记整改时间点
- 建议项：纳入后续优化，不阻断当前流程

## 5. 通过判定
- 当前审计无阻断项
- 阻断项修复后已复审通过
- 审计报告已纳入证据索引
