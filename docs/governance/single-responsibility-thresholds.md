# 单一职责判据阈值

> 目的：把“单一职责”转为可执行、可复跑、可审计的机判规则。
> 上位约束：`one law.md`、`目录框架规范.md`

## 1. 可机判阈值（强制）

### 1.1 文档文件阈值（`docs/**/*.md`）
- 阈值 D1，一级标题数量必须等于 1。
  - 检查命令：`Select-String -Path <file> -Pattern '^# ' | Measure-Object | % Count`
  - 通过条件：结果 `= 1`
- 阈值 D2，规范主题关键词组命中数必须小于等于 1 组。
  - 关键词组：`(Gate|门禁)`、`(证据|evidence)`、`(审计|audit)`、`(单一职责|职责漂移)`
  - 检查命令：对每组执行 `Select-String` 计数，再统计命中组数
  - 通过条件：命中组数 `<= 1`
- 阈值 D3，文档不得自称多主题。
  - 拦截模式：`'本文件.*(同时|覆盖|包含).*(和|与)'`
  - 通过条件：匹配数 `= 0`

### 1.2 脚本文件阈值（`tools/scripts/**/*.ps1`, `tools/githooks/*`）
- 阈值 S1，单脚本主动作类型数必须小于等于 1。
  - 动作类型集合：检查类（`test|check|validate|verify`），变更类（`set|update|remove|move|rename`）
  - 检查方法：分别统计两类关键词命中，若两类均命中则判定为混责
  - 通过条件：两类中至多一类命中
- 阈值 S2，脚本文件名必须符合“动词+对象”格式，不允许泛化命名。
  - 拦截模式：`'(^|\\)(temp|new|utils|helper)\.ps1$'`
  - 通过条件：匹配数 `= 0`

### 1.3 代码文件阈值（`src/**/*.py`）
- 阈值 C1，单文件导出的公开入口（`def run`、`class .*Tool`、`TOOL_NAME`）命中数必须小于等于 1 类职责。
  - 检查方法：按职责标签分组统计命中
  - 通过条件：职责标签组数 `<= 1`
- 阈值 C2，禁止同时出现“业务策略关键词 + 工程脚本关键词”。
  - 业务策略关键词：`friction|lyric|acoustic|prompt`
  - 工程脚本关键词：`hook|gate|consistency|bootstrap`
  - 通过条件：两组不同时命中

## 2. 职责漂移检测方法

### 2.1 变更集检测（每次提交前）
1. 收集变更文件：`git diff --name-only --cached`
2. 逐文件执行第 1 节阈值检查。
3. 输出标准化结果：
   - `PASS`：所有阈值通过
   - `DRIFT_WARN`：命中 1 条阈值
   - `DRIFT_BLOCK`：命中 >= 2 条阈值，或命中 D2/S1/C2 任一高风险阈值

### 2.2 职责漂移分级
- L1 轻度漂移：仅命中 1 条低风险阈值（D3、S2、C1）
- L2 中度漂移：命中 1 条高风险阈值（D2、S1、C2）或命中 2 条低风险阈值
- L3 重度漂移：命中 >= 2 条高风险阈值，或同一主题出现多份权威定义

### 2.3 检测结果记录
- 记录路径：`.sisyphus/evidence/task-*-sr-thresholds-check.txt`
- 必填字段：`文件`、`命中阈值`、`分级`、`处理动作`、`复检结果`

## 3. 违规处理流程（闭环）

1. 发现违规
   - 来源：pre-commit 检查、审计任务、人工抽检
   - 动作：立即标记 `DRIFT_WARN` 或 `DRIFT_BLOCK`
2. 阻断与分派
   - L1：允许继续但必须在当前任务内修复
   - L2：阻断当前 Gate，要求拆分文件或回滚后重提
   - L3：阻断合并，触发专项审计
3. 修复执行
   - 必须提交“拆分/回滚/重命名”之一的可复跑证据
4. 复检
   - 对修复后的文件重新执行第 1 节全部阈值
   - 通过条件：无阈值命中
5. 归档与追踪
   - 将本次违规与复检结果写入任务证据与治理看板
   - 连续 2 次出现 L2 或任意 1 次 L3 的模块，纳入下一轮审计优先级

## 4. 与现有治理规则衔接
- 与 `one law.md` 对齐：不满足可测试与非冗余要求时，不落库、不合并。
- 与 `目录框架规范.md` 对齐：命名、目录边界、反漂移条款直接作为阈值依据。
- 与 `docs/governance/gates.md` 对齐：`DRIFT_BLOCK` 视为 Gate 阻断项，必须修复后再进入下游。
