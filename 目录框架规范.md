# 目录框架规范（防冗余脚本版）

> 上位约束：`one law.md`  
> 开发流程：见 `开发清单.md`  
> 事实来源：`AI-music-producer PRD_v1.1.md`

---

## 1. 目标

- 统一目录职责，避免“同一功能多处实现”
- 让脚本、配置、文档可被快速定位
- 保持 Windows CLI 环境可用且可维护

## 1.1 AI-Oriented 文档结构原则（强制）

1. 文档描述必须唯一归属，禁止多处复写同一规范正文。
2. 复用规则只能以 reference 形式引用归属文档。
3. 所有规范归属关系由 `docs/ai_doc_manifest.json` 声明。
4. 合并前必须通过 `tools/scripts/check_docs_consistency.ps1`。

---

## 2. 标准目录（仓库根）

```txt
/
├─ AI-music-producer PRD_v1.1.md   # 开发指南（单一事实源）
├─ one law.md                      # 唯一法则
├─ 开发清单.md                      # TDD + hooks 执行清单
├─ 目录框架规范.md                  # 本文件
│
├─ tools/
│  ├─ githooks/                    # 版本化 hooks（pre-commit/commit-msg/pre-push）
│  └─ scripts/                     # 通用工程脚本（非业务）
│
├─ apps/                           # 应用入口（CLI 主程序）
├─ src/                            # 业务代码（按能力分域）
├─ tests/                          # 测试代码（与 src 对应）
├─ docs/                           # 设计/规范/决策文档
│
└─ projects/                       # 运行时项目目录（按 PRD）
   └─ YYYY-MM-DD_<project_name>/
      ├─ .git/
      ├─ .gitignore
      ├─ intent.md
      ├─ voice_profile.json
      ├─ reference_dna.json
      ├─ friction_report.json
      ├─ lyrics.json
      ├─ prompts/

      └─ .producer_state.sqlite
```

## 3. 防冗余脚本机制（强制）

## 3.1 新增脚本前必做

- [ ] 在 `tools/scripts/` 与 `tools/githooks/` 检索同职责脚本
- [ ] 若已有同职责脚本：**禁止新增**，必须复用或扩展
- [ ] 若确需新增：在文件头写明“为何不能复用现有脚本”

## 3.1.1 开发顺序法则（目录与脚本层）

- 开发顺序法则正文以 `one law.md` 为唯一归属，本文件仅引用执行。
- 任何新增脚本/目录前，必须完成：规则对齐 + 测试证明 + Hook 通过。

## 3.2 命名规则

- 脚本文件名必须体现动词+对象：`sync_prompts.ps1`、`validate_hooks.ps1`
- 禁止模糊命名：`temp.ps1`、`new.ps1`、`utils.ps1`、`helper.ps1`

## 3.3 职责边界

- `tools/githooks/*`：只做提交门禁相关检查
- `tools/scripts/*`：只做工程辅助，不承载业务主流程
- 业务流程实现必须在 `src/`，禁止塞进 `tools/scripts/`

---

## 4. 文件放置规则

1. 规则文档只放根目录或 `docs/`，不散落到业务目录
2. 运行产物（音频、模型缓存）不进 git
3. 测试文件与被测模块同域对应，命名可逆定位
4. 一类数据一个落点，禁止多份“同名不同内容”副本
5. 规范正文一类一处，其他文档仅允许引用该正文归属文件

---

## 5. 变更审批规则（目录层）

发生以下任意一种情况，必须先改文档再改代码：

- 新增一级目录
- 新增脚本类别
- 修改 hooks 策略
- 调整 `projects/` 运行时结构

审批检查项：

- [ ] 是否与 `one law.md` 一致
- [ ] 是否与 PRD 的 git 卫生规则一致
- [ ] 是否会引入重复执行链路

---

## 6. 快速自检（每次合并前）

- [ ] 是否新增了功能重复脚本
- [ ] 是否出现职责不清目录
- [ ] 是否把业务逻辑误放进 `tools/`
- [ ] 是否违反音频不进 git 规则

---

## 7. RULE-13 ~ RULE-18（最优解版本）

**RULE-13（命名规范）**：文件名统一可搜索、可排序、可追溯（小写/下划线或既有规范），禁止 `final_v2_new_真最终版` 一类命名。  

**RULE-14（单一输出）**：同一主题只允许一个最终输出文件；历史过程通过 git 记录，不靠重复文件保留。  

**RULE-15（反杂乱）**：根目录只允许核心入口文件；调研、脚本、草稿不得长期驻留根目录。  

**RULE-16（反漂移）**：文档规则、Manifest、Hook 行为必须一致；若冲突，以 `one law.md` 与 Manifest 为准并立即修正。  

**RULE-17（最小复杂度）**：优先 JSON + 规则 + 轻量脚本。任何新增能力先走低依赖方案，再评估重方案。  

**RULE-18（可安装性）**：所有新增能力必须能进入 Upsert → ZIP → Install 流程，且可在新环境一键验证。  

---

## 8. 常见违规案例（精简）

- 把文件写到 `C:\` 或用户桌面。
- 新建 `manifest_v2.json`、`plan_new.md` 等重复权威文件。
- 产生 `nul`、临时调研目录、一次性脚本后不清理。
- Hook 命名与清单不一致（文档写一套，代码跑另一套）。
- 修改知识库后不做 Schema / pytest 校验直接进入下游。

---

## 9. 违规处理流程（Violation Handling）

1. 立即停止当前 Task。  
2. 记录违规类型、影响路径、责任模块。  
3. 清理异常产物并回滚到最近可验证状态。  
4. 重新执行 pre_hook，验证通过后再继续。
