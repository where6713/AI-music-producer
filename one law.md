# ONE LAW（唯一法则）

## 唯一法则

**凡新增内容，必须同时满足三条：**

1. **能追溯到 PRD 条款**（先有需求条款，再有代码与脚本）
2. **能被测试证明**（先写失败测试，再写最小实现，再重构）
3. **不制造冗余维护面**（不新增重复脚本、重复目录、重复流程）

> 不满足任意一条：不落库、不合并。

## 开发顺序法则

**先法则，后 Hook；先测试，后实现；先通过 Hook，再允许流程扩张。**

- 法则未定义或冲突：停止实现
- 测试未先失败：禁止进入实现
- Hook 未通过：禁止继续新增流程/脚本/目录

---

## 解释（用于团队统一口径）

- 本仓库以 `AI-music-producer PRD_v1.1.md` 为开发指南与约束来源。
- “OOST 文件管理原则”在本项目按 **One Source of Truth + 保持简洁** 执行：
  - 单一事实源（PRD / 规范文档）
  - 单一职责文件（一个文件只解决一类问题）
  - 单一路径执行（同一流程不允许两套脚本）

## AI-Oriented 文档法（防抽样漂移）

- 文档构建以 **AI 可检索一致性** 为优先，不以长篇人类叙述为优先。
- 每个规范主题必须有唯一归属文件（canonical owner）。
- 非归属文件只能写“引用（reference）”，不得复制规范正文。
- 统一由 `docs/ai_doc_manifest.json` + `tools/scripts/check_docs_consistency.ps1` 做脚本一致性校验。

---

## 禁止项（红线）

- 禁止未在 PRD 或治理文档登记的“顺手脚本”
- 禁止把二进制音频纳入 git 版本对象
- 禁止跳过测试直接提交实现
- 禁止在本地与 CI 维护两套不同质量标准
- 禁止“同义不同名”的重复目录（如 `tools/` 与 `scripts/` 混用承载同类职责）

---

## 合并门槛

一次变更要进入主线，必须满足：

- [ ] 关联 PRD 条款（在提交说明中标注条款）
- [ ] 对应测试已新增且先失败后通过（TDD 证据）
- [ ] 通过 hooks 与 CI 校验
- [ ] 未新增冗余脚本/目录
