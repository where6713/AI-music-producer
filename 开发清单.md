### 0.1 开发顺序法则（强制）

## 1. TDD 清单（强制执行）

### 1.1 Red（先写失败测试）
- [ ] 先写测试，不先写实现
- [ ] 运行测试，确认**因断言失败**而失败（不是因为语法错误/环境错误）

### 1.2 Green（最小实现通过）
- [ ] 仅实现让当前测试通过的最小代码
- [ ] 不顺手加“未来可能用到”的逻辑

### 1.3 Refactor（重构不改行为）
- [ ] 仅在测试全绿前提下重构
- [ ] 重构后再次运行测试，结果必须全绿

### 1.4 完成门槛
- [ ] 本次变更对应测试存在且通过
- [ ] 无新增回归
- [ ] 关键路径可复现（命令与结果可被他人复跑）

## 2. Git 与 Hooks 规范（强制）

## 2.1 提交前（pre-commit）
- [ ] 仅提交必要文件（`git status` 检查）
- [ ] 基础格式与静态检查通过
- [ ] 禁止提交密钥/令牌/凭证
- [ ] 禁止提交二进制音频（遵守 PRD `.gitignore` 规则）

## 2.2 提交信息（commit-msg）
- [ ] 使用约定格式：`type(scope): summary`
- [ ] summary 清晰描述“为什么改”

## 2.3 推送前（pre-push）
- [ ] 运行快速测试集（至少覆盖本次改动）
- [ ] 本地结果与 CI 口径一致

## 2.4 Hook 目录与安装方式（默认）

采用**无额外框架锁定**方案：

```txt
tools/
  githooks/
    pre-commit
    commit-msg
    pre-push
```

一次性安装：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "tools/scripts/install_hook_chain.ps1"
```

安装后自检：

```powershell
git config --get core.hooksPath
```

期望输出：`tools/githooks`

## 2.6 no-bypass 策略（P01.02）

- [ ] 禁止在常规开发中使用 `--no-verify` 跳过 `pre-commit` / `commit-msg` / `pre-push`
- [ ] `post-commit` 会记录提交是否经过完整 hook 链；`pre-push` 会阻断未验证提交
- [ ] 若历史提交被识别为未验证，必须执行 `git commit --amend --no-edit` 重新走 hook 链
- [ ] 仅在治理审批的 break-glass 场景允许临时绕过，并在证据中登记原因与修复动作

> 若后续确需引入 Husky/Lefthook，必须先在 PRD/治理文档登记“为什么需要”，否则不引入。
