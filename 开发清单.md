## 0. 启动前检查（必须）

### 0.1 开发顺序法则（强制）

- [ ] 先法则，后 Hook
- [ ] 先测试，后实现
- [ ] 先通过 Hook，再允许流程扩张

- [ ] 本次变更能明确对应到 PRD 条款（在提交说明中写明）
- [ ] 不触犯四条红线：不造轮子 / 不浪费抽卡 / 不暴露 traceback / 不过度工程
- [ ] 不引入 PRD 明确排除项（例如 ViSQOL + FAD）
- [ ] 遵循《目录框架规范.md》RULE-13~18（命名/单一输出/反杂乱/反漂移/最小复杂度/可安装性）

## 1. TDD 清单（强制执行）

### 1.1 Red（先写失败测试）
- [ ] 先写测试，不先写实现
- [ ] 运行测试，确认**因断言失败**而失败（不是因为语法错误/环境错误）

### 1.2 Green（最小实现通过）
- [ ] 仅实现让当前测试通过的最小代码
- [ ] 不顺手加“未来可能用到”的逻辑

### 1.3 Refactor（重构不改行为）
- [ ] 仅在测试全绿前提下重构
- [ ] 重构后再次运行测试，结果必须全绿

### 1.4 完成门槛
- [ ] 本次变更对应测试存在且通过
- [ ] 无新增回归
- [ ] 关键路径可复现（命令与结果可被他人复跑）

---

## 2. Git 与 Hooks 规范（强制）

## 2.1 提交前（pre-commit）
- [ ] 仅提交必要文件（`git status` 检查）
- [ ] 基础格式与静态检查通过
- [ ] 禁止提交密钥/令牌/凭证
- [ ] 禁止提交二进制音频（遵守 PRD `.gitignore` 规则）

## 2.2 提交信息（commit-msg）
- [ ] 使用约定格式：`type(scope): summary`
- [ ] summary 清晰描述“为什么改”

## 2.3 推送前（pre-push）
- [ ] 运行快速测试集（至少覆盖本次改动）
- [ ] 本地结果与 CI 口径一致

## 2.4 Hook 目录与安装方式（默认）

采用**无额外框架锁定**方案：

```txt
tools/
  githooks/
    pre-commit
    commit-msg
    pre-push
```

一次性安装：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "tools/scripts/install_hook_chain.ps1"
```

安装后自检：

```powershell
git config --get core.hooksPath
```

期望输出：`tools/githooks`

## 2.6 no-bypass 策略（P01.02）

- [ ] 禁止在常规开发中使用 `--no-verify` 跳过 `pre-commit` / `commit-msg` / `pre-push`
- [ ] `post-commit` 会记录提交是否经过完整 hook 链；`pre-push` 会阻断未验证提交
- [ ] 若历史提交被识别为未验证，必须执行 `git commit --amend --no-edit` 重新走 hook 链
- [ ] 仅在治理审批的 break-glass 场景允许临时绕过，并在证据中登记原因与修复动作

> 若后续确需引入 Husky/Lefthook，必须先在 PRD/治理文档登记“为什么需要”，否则不引入。

## 2.5 目录唯一性（新增）

- [ ] 当前开发仅在 `G:\AI-music-producer` 内进行
- [ ] 隔离开发优先使用分支（`git checkout -b ...`），不创建仓库外 worktree 目录
- [ ] 若发现历史遗留外部开发副本目录，先清理再继续开发

## 3. 变更提交清单（每次都走）

- [ ] 需求映射：PRD 条款已标注
- [ ] TDD 证据：先失败后通过
- [ ] Hook 通过：pre-commit / commit-msg / pre-push
- [ ] 文档同步：涉及规则变更已更新治理文档
- [ ] 无冗余脚本：未新增重复功能脚本

---

## 4. 禁止清单（直接驳回）

- [ ] 绕过测试直接实现
- [ ] 使用 `--no-verify` 常态化绕过 hooks
- [ ] 将 `post-commit` / `pre-push` 的未验证阻断告警视为可忽略信息
- [ ] 一次提交混入无关改动
- [ ] 新增与现有脚本功能重复的“临时脚本”
- [ ] 提交 PRD 明确排除的能力或依赖

---

## 5. 最小执行顺序（团队统一）

1. 先确认法则与规则（`one law.md` + 本清单）  
2. 先写失败测试  
3. 再写最小实现  
4. 跑测试 + hooks（全部通过）  
5. 通过后才允许新增流程/脚本/目录  
6. 小步提交（原子提交）  
7. 更新必要文档
